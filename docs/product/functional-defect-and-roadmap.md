> **版本**：v0.1
> **编写日期**：2026-06-16
> **视角**：从「客户为什么买单」出发，而非「我们有什么技术」出发。
> **配套文档**：[需求文档](../../requirement.md) · [PRD v0.1](../../麒麟安全智能运维%20Agent%20PRD%20v0.1.md) · [MVP 路线 v0.1](../../MVP%20功能优先级与版本路线%20v0.1.md)

## 0. 摘要

| 项 | 数量 |
|---|---|
| P0 级缺陷（影响演示与核心差异化） | 4 项 |
| P1 级缺陷（影响竞争力与企业级落地） | 6 项 |
| P2 级缺陷（影响规模化与体验） | 6 项 |
| 短期迭代方向（≤3 个月） | 3 项 |
| 中期迭代方向（6~9 个月） | 3 项 |
| 长期生态建设方向（9~18 个月） | 6 项 |

**核心结论**：
1. **演示交付**：缺演示 PPT 与 ≤7 分钟视频，对应大赛 20% 评分项，必须立即补齐。
2. **核心差异化**：根因分析缺乏结构化产出，是 requirement.md §18「智能化根因分析」评分项的最大缺口。
3. **企业采购门槛**：缺多主机集群、告警通道、审计归档三项能力，阻碍进入大型企业采购清单。
4. **客户粘性**：缺知识库与插件市场，6 个月内可能被通用 ChatGPT 类产品替代。

## 一、功能层面缺陷报告

缺陷按「业务影响优先级」而非「技术严重程度」排序。每一项都从客户视角描述痛点，而非技术描述。

### 🔴 P0 — 影响演示交付与核心差异化（必须立即修复）

#### D-01 根因分析缺乏结构化产出

- **缺陷描述**：`AgentResponseBuilder` 把根因推断隐式拼进 Markdown 文本，前端只展示一段自然语言描述，看不到「为什么这么判断」。
- **客户痛点**：
  - 评委问"你凭什么判断是大文件占满？"——只能嘴上讲，不能可视化演示。
  - 合规审计需要"Agent 决策依据"，当前是黑盒。
- **影响维度**：requirement.md §18「智能化根因分析能力」评分项的最大缺口。
- **修复成本**：约 2~3 天（数据模型 + 前端组件 + 演示场景 2 重做）。

#### D-02 `lsof` 能力缺位

- **缺陷描述**：11 个工具覆盖基础运维，但「进程卡死 → 是哪个文件被锁」这类根因场景无法定位。`ProcessDetailTool` 只看进程本身，看不到 fd/socket。
- **客户痛点**：
  - SRE 实际工作中 80% 的「为什么挂」都跟 fd/socket 有关（文件锁、端口占用、磁盘 IO 阻塞）。
  - requirement.md 明确点名 `lsof` 作为环境感知命令之一。
- **影响维度**：requirement.md §18「OS 感知与 MCP 插件实现」完整性。
- **修复成本**：约半天（`lsof -p <pid>` 集成进 `ProcessDetailTool` 或新增独立工具）。

#### D-03 演示 PPT + 演示视频未交付

- **缺陷描述**：大赛硬交付物第 8、9 项，仓库内未找到 .pptx 与 .mp4 文件。演示脚本 v0.1.md（6:30）已就位，但视频未录制。
- **客户痛点**：
  - 评委无法在 7 分钟内感知产品全貌。
  - 演示视频是 requirement.md 评分项「文档与演示 20%」的核心载体。
- **影响维度**：直接丢失约 15~20% 的评分。
- **修复成本**：PPT 约 1 天；视频约半天录制 + 后期。

#### D-04 LLM 调用失败的兜底口径过窄

- **缺陷描述**：当 LLM 完全下线（断网、限流、API Key 失效）时，`HybridIntentService` 退化为纯规则匹配，意图识别准确率从 ~90% 跌到 ~60%。"重启数据库""查慢 SQL"等未匹配规则的请求直接拒绝服务。
- **客户痛点**：
  - 真实部署在 LoongArch + 隔离网络的机房很常见（不允许直连公网 LLM），产品易"趴窝"。
  - "AI Agent 离线即不可用"对企业是致命的——传统监控至少还能跑。
- **影响维度**：产品 SLA 与可靠性。
- **修复成本**：约 1~2 天（扩规则库 + 增加离线 FAQ 库）。

### 🟠 P1 — 影响竞争力与企业级落地（3 个月内修复）

#### D-05 多主机管理缺位

- **缺陷描述**：当前是单机 Agent。客户实际管理 50~500 台 Kylin 主机，每台开一个 ChatConsole 不现实。
- **客户痛点**："我管 200 台机器，难道开 200 个窗口？"——只能做 POC 产品。
- **影响维度**：真实场景落地受限。
- **修复成本**：约 1~2 个月（引入 Agent Node + Control Plane 架构）。

#### D-06 告警推送通道缺位

- **缺陷描述**：L4 阻断事件、安全注入事件只能被动看 `SecurityCenter`，无主动推送。
- **客户痛点**：真实 SOC 要求 7×24 告警推送（钉钉/企微/飞书/邮件/Webhook）。当前响应延迟从"秒级"变成"早会才发现"。
- **影响维度**：安全价值归零；定价上不去。
- **修复成本**：约 1~2 周（Webhook 抽象 + 4 个通道适配器）。

#### D-07 变更无回滚

- **缺陷描述**：`safe_service_restart` 等执行动作失败后，Agent 只报"启动失败"，没有自动回滚到上一版本配置。
- **客户痛点**：SRE 救火时本来就紧张，还要手动 `git stash` 或备份恢复——MTTR 翻倍。
- **影响维度**：高风险动作的"敢用度"。
- **修复成本**：约 2~3 周（引入快照机制 + 回滚编排）。

#### D-08 无定时巡检 / 无人值守

- **缺陷描述**：当前产品只能"用户问 → Agent 答"，没有"每天 8:00 自动跑一遍健康检查并推送日报"。
- **客户痛点**：与"传统 Zabbix + 告警"无差异化——客户会说"那我为啥不用 Zabbix？"
- **影响维度**：错失 AI Ops 区别于传统监控的关键差异点。
- **修复成本**：约 2~3 周（cron-like 调度器 + 巡检模板）。

#### D-09 无知识库 / 企业 RAG

- **缺陷描述**：LLM 用通用知识答 OS 问题，不接企业私有知识。
- **客户痛点**：客户问"我们公司的 nginx upstream 配置规范"，答不上——所有诊断经验 6 个月后流失。
- **影响维度**：客户粘性、行业知识沉淀。
- **修复成本**：约 3~4 周（向量库 + 检索增强 + 与 HybridIntentService 集成）。

#### D-10 报告生成是模板拼装

- **缺陷描述**：`ReportService` 把数据塞进固定模板，无法做"跨多个 session 的故障复盘"（如"过去 7 天所有 nginx 异常的相关性"）。
- **客户痛点**：大型故障复盘无能为力。
- **影响维度**：高管汇报、复盘会议场景。
- **修复成本**：约 2~3 周（跨 session 关联分析 + 时间窗口聚合）。

### 🟡 P2 — 影响规模化与体验（6~12 个月规划）

| # | 缺陷 | 业务痛点 | 修复成本 |
|---|---|---|---|

| **D-12** | 运维剧本（Workflow）编排缺位 | "磁盘满 → 自动清缓存 → 自动健康检查 → 自动报告"——目前只能人问一句。 | 4~6 周 |

## 二、未来迭代路线图

按 **业务价值 × 实施难度** 排序。每一项都给出：现状痛点 → 解决思路 → 业务价值（量化）→ 目标用户。

### 第一档：高价值 + 低/中难度 → **3 个月内交付**

#### 方向 1：RCA 推理链结构化（修复 D-01）

- **现状痛点**：根因隐藏在 Markdown 文本里，评委和客户都看不到推理路径。
- **解决思路**：
  - `Report` 模型加 `RootCauseChain` 字段（`symptom / evidence[] / hypothesis[] / conclusion / confidence`）
  - 前端 `ReportPreview` 增加 `<ReasoningChain>` 组件，可视化展示「工具证据 → 假设 → 排除 → 结论」
  - 演示场景 2（磁盘诊断）改为"先看 86% → 扫大文件 → 排除 /var/lib/mysql → 锁定 /var/log/app.log = 12GB"
- **业务价值**：
  - 演示加分项（直接对应 requirement.md §18 评分维度）
  - 故障复盘时间从「翻聊天记录 30 分钟」降到「看结构化字段 3 分钟」
  - 合规审计：可向等保检查员提供「Agent 决策依据」，从 AI 黑盒变成 AI 白盒
- **目标用户**：SRE 主管、CTO、合规审计员
- **量化指标**：故障复盘 MTTR ↓ 80%

#### 方向 2：工具扩到 ≥ 14 个，lsof 集成 + 资产清单（修复 D-02、D-11）

- **现状痛点**：11 个工具覆盖基础运维，"为什么这个进程僵死"答不上；服务器资产盘点要人工写脚本。
- **解决思路**：
  - `LsofTool`：枚举进程 fd、网络 socket、内存映射
  - `OpenFilesTool`：按目录/扩展名聚合
  - `PackageListTool` / `ServiceDiscoveryTool`：自动盘点资产
  - 新增"资产清单"页面（`/assets`），汇总所有主机的软件/端口/服务
- **业务价值**：
  - 存量运维客户最关心的问题之一：「我有多少台机器跑着过期版本的 OpenSSL？」——之前要写脚本扫，现在 Agent 一句话回答
  - 安全前置：发现未授权开放的端口 → 立即触发风险告警
  - 新机器入网验收：从「半天手工盘点」降到「5 分钟自动报告」
- **目标用户**：安全工程师、运维经理、新机器入网流程负责人
- **量化指标**：资产盘点耗时 ↓ 90%

#### 方向 3：告警通道（修复 D-06）

- **现状痛点**：安全事件只能查 `SecurityCenter`，无主动推送。
- **解决思路**：
  - 新增 `NotificationChannel` 配置（钉钉/企微/飞书/邮件/Webhook）
  - L4 BLOCK / Prompt Inject 命中 / L2 拒绝确认等事件实时推送
  - 配置粒度：按事件类型、按风险等级
- **业务价值**：
  - **安全运维的核心 KPI 是 MTTD（Mean Time To Detect）**——从「早会才发现」降到「秒级告警」
  - 客户愿为这个能力付溢价（参考 SOC/SIEM 行业标准）
  - 差异化卖点：「AI Agent 不只是事后查，还能事前/事中告警」
- **目标用户**：SOC 分析师、信息安全主管
- **量化指标**：安全事件 MTTD ↓ 95%

---

### 第二档：高价值 + 高难度 → **6~9 个月规划**

#### 方向 4：多主机 Agent 集群（修复 D-05）

- **解决思路**：
  - 引入 `Agent Node` 概念：每台 Kylin 主机部署轻量 Agent，上报给中央 `Control Plane`
  - `ChatConsole` 支持「@主机名」或「在所有 web 服务器上…」类指令
  - 中央节点统一审计、统一报告
- **业务价值**：
  - **从"POC 产品"变"生产级产品"**——可对标 Datadog Agent、Prometheus node_exporter
  - 客户的采购决策单位是"机房"而不是"机器"，集群化后才能进入采购清单
  - 预估市场规模扩大 10~100 倍（从单机 POC 到企业级批量采购）
- **目标用户**：中大型企业 IT 部门、云服务商
- **量化指标**：单运维人员可管理主机数 ↑ 50 倍

#### 方向 5：无人值守巡检 + Workflow 编排（修复 D-08、D-12）

- **解决思路**：
  - 定时任务：cron-like 配置（每天/每周/每月）
  - 巡检模板：内置 health/disk/log/service 4 种巡检剧本
  - Workflow DSL：可视化编排「条件 → 动作 → 通知」链路（如「磁盘 > 85% → 自动清缓存 → 不恢复则告警 → 通知 SRE」）
- **业务价值**：
  - **从"问答工具"变成"虚拟 SRE 同事"**——这是与 ChatGPT/Copilot 的根本差异
  - 中型企业可节省 1~2 个初级 SRE 人力成本（按 15~25 万/年算，3~6 个月回本）
  - 客户续费率高：沉淀的 Workflow 是迁移成本
- **目标用户**：中型企业 IT 主管（CIO 直接拍板）
- **量化指标**：初级 SRE 人力需求 ↓ 30~50%

#### 方向 6：知识库 / 企业 RAG（修复 D-09）

- **解决思路**：
  - 私有知识库：上传公司 OS 配置规范、应急预案、历史故障复盘文档
  - 向量化存储 + 检索增强生成
  - 与 `HybridIntentService` 集成：规则 → LLM → RAG 三段式
- **业务价值**：
  - **「为什么用你这家而不是通用 ChatGPT？」**——因为接入了企业知识
  - 行业知识库可作为增值产品（金融/教育/医疗行业包，单独定价）
  - 客户切换成本显著提高（沉淀 6 个月后，没人愿意换）
- **目标用户**：金融、医疗等强合规行业 IT 主管
- **量化指标**：客户 6 个月续费率 ↑ 40%

### 第三档：中长期生态建设 → **9~18 个月**

| 方向 | 业务价值 | 对应缺陷 | 量化指标 |
|---|---|---|---|
| **变更回滚 / 灰度执行** | 客户最关心「敢不敢用」——可回滚 = 敢用；MTTR 减半 | D-07 | 服务故障 MTTR ↓ 50% |
| **多租户 + 分账** | SaaS 化前提；按团队/部门独立审计 | D-14 | SaaS 化收入占比可达 60% |
| **审计日志归档策略** | 满足等保 2.0 强制要求；进入合规采购清单 | D-15 | 进入金融/政府合规采购清单 |

| **异常行为基线（UEBA）** | 学 7 天正常行为 → 偏离即告警（User & Entity Behavior Analytics） | 新增 | 高级威胁检出率 ↑ 70% |

## 四、给项目的建议（**不要炫技**）

### ✅ 应该做（业务驱动）

1. **先补 RCA 结构化**——这是 requirement.md §18「智能化根因分析」评分项的硬要求，也是合规审计的命门。
2. **做告警通道**——把「事后查」变成「事中告警」，客单价可翻 2~3 倍。
3. **做多主机集群**——进入企业采购清单的前提。
4. **做演示 PPT + 视频**——大赛 20% 评分项。

### ❌ 不要做（炫技陷阱）

- **「支持 20 种国产 LLM」**——客户只用 1~2 种，扩到 5 种边际价值为零。
- **「多模态输入（语音/图片）」**——SRE 不会对着服务器拍照片。
- **「支持 50 种 OS」**——题目锁死 Kylin V11，其他是噪声。
- **「AI 自动选择最优工具链」**——客户要的是**可解释**而非「最优」。
- **「自研向量数据库」**——直接用 pgvector 或 Milvus，差异化在场景而非底层。
- **「自定义编程语言写 Workflow DSL」**——YAML + 表达式引擎足矣。

> **核心原则**：每一项新功能上线前问一句——「客户因为这个功能多付 1 万还是多付 10 万？」答不上来就不做。
