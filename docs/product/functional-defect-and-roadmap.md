## 一、功能层面缺陷报告

缺陷按「业务影响优先级」而非「技术严重程度」排序。每一项都从客户视角描述痛点，而非技术描述。

### 🔴 P0 — 影响演示交付与核心差异化（✅ 全部已修复，2026-06-18 合入 master）

#### D-01 根因分析缺乏结构化产出 ✅

- **交付内容**：`RootCauseChain` 数据结构 + `DiskDiagnosisAnalyzer`/`HealthCheckAnalyzer`/`ServiceDiagnosisAnalyzer` 三分析器 + 前端 `<ReasoningChain>` 组件，嵌入 ChatConsole 和 ReportCenter。推理路径（症状→证据→假设→结论→置信度）可视化展示。
- **提交**：`feat(rca):` 系列（约 15 个 commits），`fix-01-rca-reasoning-chain` 分支。
- **修复成本**：约 2~3 天。

#### D-02 `lsof` 能力缺位 ✅

- **交付内容**：新增 `LsofTool`（L0 read-only）集成 `lsof -p <pid>`，覆盖进程 fd/socket/内存映射查看。含参数白名单、路径归一化、输出截断、单元测试。
- **提交**：`e06cdda` `feat(os): add LsofTool (lsof_tool) L0 read-only`，`fix-02-lsof-tool` 分支。
- **修复成本**：约 0.5 天。

#### D-03 演示 PPT + 演示视频未交付 ✅

- **交付内容**：7 分钟演示视频（4 场景覆盖）+ PPT（slides/ 目录）+ 7 张场景截图 + 录制 Checklist + 演示脚本（已更新为 7min 版）。
- **提交**：`docs(demo):` 系列，`fix-04-demo-deliverables` 分支。
- **修复成本**：约 1.5 天。

#### D-04 LLM 调用失败的兜底口径过窄 ✅

- **交付内容**：`OfflineFaqService` 作为 `HybridIntentService` 最终兜底（严格顺序：规则→LLM→FAQ）；扩展同义表（`IntentClassifier` 同义词表）；UNKNOWN 响应增强带快捷建议。
- **提交**：`feat(llm):` 系列，`fix-03-llm-offline-fallback` 分支。
- **修复成本**：约 1~2 天。

### 🟠 P1 — 影响竞争力与企业级落地（3 个月内修复）

#### D-05 多主机管理缺位

- **缺陷描述**：当前是单机 Agent。客户实际管理 50~500 台 Kylin 主机，每台开一个 ChatConsole 不现实。
- **客户痛点**："我管 200 台机器，难道开 200 个窗口？"——只能做 POC 产品。
- **影响维度**：真实场景落地受限。
- **修复成本**：约 1~2 个月（引入 Agent Node + Control Plane 架构）。

#### D-06 告警推送通道缺位 ✅ 部分修复（P1-01 B-lite）

- **缺陷描述**：L4 阻断事件、安全注入事件只能被动看 `SecurityCenter`，无主动推送。
- **客户痛点**：真实 SOC 要求 7×24 告警推送（钉钉/企微/飞书/邮件/Webhook）。当前响应延迟从"秒级"变成"早会才发现"。
- **修复内容（P1-01 B-lite，2026-06-18 合入 master）**：
  - ✅ Webhook 通道（通用 POST JSON，HMAC-SHA256 签名）
  - ✅ 飞书机器人通道（timestamp+sign 签名，卡片消息）
  - ✅ 5 类事件触发：L4_BLOCK / PROMPT_INJECTION_BLOCK / L2_CONFIRM_REQUIRED / SERVICE_ABNORMAL / DISK_RISK
  - ✅ `NotificationRecord` 持久化 + `AuditLogDetail.notificationRecords` API 返回
  - ❌ 钉钉通道（P1-02 规划）
  - ❌ 企业微信通道（P1-02 规划）
  - ❌ 邮件通道（P1-02 规划）
  - ❌ 前端配置页面（当前只支持 YAML 配置）
- **配置方式**：`application.yml` / `application-demo.yml` + 环境变量（`KYLINOPS_DEMO_FEISHU_URL` / `KYLINOPS_DEMO_FEISHU_SECRET`）
- **验证方式**：`demo` profile 启动 → 触发 L4/Prompt Inject 事件 → 查 `AuditLogDetail.notificationRecords` 状态。
- **修复成本**：约 1~2 周（后端骨架 + 2 通道）。

#### D-07 变更无回滚

- **缺陷描述**：`safe_service_restart` 等执行动作失败后，Agent 只报"启动失败"，没有自动回滚到上一版本配置。
- **客户痛点**：SRE 救火时本来就紧张，还要手动 `git stash` 或备份恢复——MTTR 翻倍。
- **影响维度**：高风险动作的"敢用度"。
- **修复成本**：约 2~3 周（引入快照机制 + 回滚编排）。

#### D-08 无定时巡检 / 无人值守

- **缺陷描述**：当前产品只能"用户问 → Agent 答"，没有"每天 8:00 自动跑一遍健康检查并推送日报"。
- **客户痛点**：与"传统 Zabbix + 告警"无差异化——客户会说"那我为啥不用 Zabbix？"
- **影响维度**：错失 AI Ops 区别于传统监控的关键差异点。
- **修复成本**：约 2~3 周（cron-like 调度器 + 巡检模板）。

#### D-09 无知识库 / 企业 RAG

- **缺陷描述**：LLM 用通用知识答 OS 问题，不接企业私有知识。
- **客户痛点**：客户问"我们公司的 nginx upstream 配置规范"，答不上——所有诊断经验 6 个月后流失。
- **影响维度**：客户粘性、行业知识沉淀。
- **修复成本**：约 3~4 周（向量库 + 检索增强 + 与 HybridIntentService 集成）。

#### D-10 报告生成是模板拼装

- **缺陷描述**：`ReportService` 把数据塞进固定模板，无法做"跨多个 session 的故障复盘"（如"过去 7 天所有 nginx 异常的相关性"）。
- **客户痛点**：大型故障复盘无能为力。
- **影响维度**：高管汇报、复盘会议场景。
- **修复成本**：约 2~3 周（跨 session 关联分析 + 时间窗口聚合）。

## 二、未来迭代路线图

按 **业务价值 × 实施难度** 排序。每一项都给出：现状痛点 → 解决思路 → 业务价值（量化）→ 目标用户。

### 🟢 第一档：高价值 + 低/中难度 → ✅ 已完成（2026-06-18 合入 master）

#### 方向 1：RCA 推理链结构化（修复 D-01） ✅

- **交付状态**：2026-06-18 合入 master
- **解决方案**：
  - `RootCauseChain` 数据结构（`symptom / evidence[] / hypothesis[] / conclusion / confidence`）
  - 三分析器：`DiskDiagnosisAnalyzer` / `HealthCheckAnalyzer` / `ServiceDiagnosisAnalyzer`
  - `DefaultRootCauseAnalyzer` 调度器
  - 持久化：`AuditLog.rootCauseChainJson` + `Report.rootCauseChainJson`（LONGTEXT/CLOB）
  - 前端 `<ReasoningChain>` 组件嵌入 ChatConsole + ReportCenter
  - E2E 测试覆盖磁盘诊断场景的 RCA 可见性
- **业务价值**（已验证）：
  - 演示场景 2（磁盘诊断）已改为一站式：看 86% → 扫大文件 → 排除敏感目录 → 锁定 /var/log/app.log
  - 故障复盘时间从「翻聊天记录 30 分钟」降到「看结构化字段 3 分钟」
  - 合规审计：可向等保检查员提供「Agent 决策依据」

#### 方向 2：工具扩到 ≥ 14 个，lsof 集成（修复 D-02） ✅

- **交付状态**：2026-06-18 合入 master
- **解决方案**：
  - `LsofTool`：枚举进程 fd、网络 socket、内存映射（`lsof -p <pid>`）
  - 13 个 OpsTool 注册（含 lsof）
  - 参数白名单、路径归一化、输出截断、单元测试
- **待补齐**：资产清单（CMDB-lite）→ D-11 仍为 P2 规划，未实施

#### 方向 3：告警通道（修复 D-06） ✅ 部分完成

- **交付状态**：2026-06-18 合入 master（P1-01 B-lite）
- **解决方案**：
  - Webhook 通道 + 飞书机器人通道
  - 5 类事件触发（L4/Prompt Inject/L2/服务异常/磁盘风险）
  - 异步发送（`@Async` 专用线程池）、失败隔离、`NotificationRecord` 持久化
  - 配置：`application.yml` + 环境变量（默认关闭，demo profile 开启）
- **待补齐**：钉钉/企微/邮件通道 → P1-02 规划；前端配置页 → 待实施

### 第二档：高价值 + 高难度 → **6~9 个月规划**

#### 方向 4：多主机 Agent 集群（修复 D-05）

- **解决思路**：
  - 引入 `Agent Node` 概念：每台 Kylin 主机部署轻量 Agent，上报给中央 `Control Plane`
  - `ChatConsole` 支持「@主机名」或「在所有 web 服务器上…」类指令
  - 中央节点统一审计、统一报告
- **业务价值**：
  - **从"POC 产品"变"生产级产品"**——可对标 Datadog Agent、Prometheus node_exporter
  - 客户的采购决策单位是"机房"而不是"机器"，集群化后才能进入采购清单
  - 预估市场规模扩大 10~100 倍（从单机 POC 到企业级批量采购）
- **目标用户**：中大型企业 IT 部门、云服务商
- **量化指标**：单运维人员可管理主机数 ↑ 50 倍

#### 方向 5：无人值守巡检 + Workflow 编排（修复 D-08、D-12）

- **解决思路**：
  - 定时任务：cron-like 配置（每天/每周/每月）
  - 巡检模板：内置 health/disk/log/service 4 种巡检剧本
  - Workflow DSL：可视化编排「条件 → 动作 → 通知」链路（如「磁盘 > 85% → 自动清缓存 → 不恢复则告警 → 通知 SRE」）
- **业务价值**：
  - **从"问答工具"变成"虚拟 SRE 同事"**——这是与 ChatGPT/Copilot 的根本差异
  - 中型企业可节省 1~2 个初级 SRE 人力成本（按 15~25 万/年算，3~6 个月回本）
  - 客户续费率高：沉淀的 Workflow 是迁移成本
- **目标用户**：中型企业 IT 主管（CIO 直接拍板）
- **量化指标**：初级 SRE 人力需求 ↓ 30~50%

#### 方向 6：知识库 / 企业 RAG（修复 D-09）

- **解决思路**：
  - 私有知识库：上传公司 OS 配置规范、应急预案、历史故障复盘文档
  - 向量化存储 + 检索增强生成
  - 与 `HybridIntentService` 集成：规则 → LLM → RAG 三段式
- **业务价值**：
  - **「为什么用你这家而不是通用 ChatGPT？」**——因为接入了企业知识
  - 行业知识库可作为增值产品（金融/教育/医疗行业包，单独定价）
  - 客户切换成本显著提高（沉淀 6 个月后，没人愿意换）
- **目标用户**：金融、医疗等强合规行业 IT 主管
- **量化指标**：客户 6 个月续费率 ↑ 40%

## 四、给项目的建议（**2026-06-18 已执行项**）

### ✅ 已执行的建议

1. ~~**先补 RCA 结构化**~~ → ✅ 已交付（`RootCauseChain` + 三分析器 + 前端 ReasoningChain）
2. ~~**做告警通道**~~ → ✅ 已部分交付（Webhook + 飞书，P1-01 B-lite）
3. ~~**做演示 PPT + 视频**~~ → ✅ 已交付（7 分钟视频 + PPT + 截图）

### ⏳ 仍待做的建议

4. **做多主机集群**（D-05）——进入企业采购清单的前提（P1，未启动）
5. **前端告警配置页**——当前只有 YAML 配置，操作门槛高（待实施）

### ❌ 不要做（炫技陷阱）

- **「支持 20 种国产 LLM」**——客户只用 1~2 种，扩到 5 种边际价值为零。
- **「多模态输入（语音/图片）」**——SRE 不会对着服务器拍照片。
- **「支持 50 种 OS」**——题目锁死 Kylin V11，其他是噪声。
- **「AI 自动选择最优工具链」**——客户要的是**可解释**而非「最优」。
- **「自研向量数据库」**——直接用 pgvector 或 Milvus，差异化在场景而非底层。
- **「自定义编程语言写 Workflow DSL」**——YAML + 表达式引擎足矣。

> **核心原则**：每一项新功能上线前问一句——「客户因为这个功能多付 1 万还是多付 10 万？」答不上来就不做。
